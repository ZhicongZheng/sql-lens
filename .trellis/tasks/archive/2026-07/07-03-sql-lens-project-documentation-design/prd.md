# SQL Lens Project Documentation Design

## Goal

Create the first complete design and documentation set for SQL Lens, a developer-first SQL debug proxy whose core promise is:

> See the SQL your application actually executes.

The deliverable is documentation only. No backend, frontend, protocol, storage, or build-system business code should be written in this task.

## Background

SQL Lens is intended to become the SQL-protocol equivalent of Charles, Fiddler, or mitmproxy: a transparent debugging and observability proxy for databases.

Applications should only need to change their database address:

```text
Application -> SQL Lens -> Database
```

The repository currently contains only initial project instructions and a seed PRD file. There is no application code yet.

The source `SQL_Lens_PRD.md` now uses the canonical name "SQL Lens" and explicitly defines the product as a multi-protocol SQL debug proxy. MySQL-compatible databases are the first implementation target, not the long-term product boundary.

## Target Users

- Backend engineers debugging SQL generated by applications, ORMs, and drivers.
- Database engineers investigating slow or failed SQL in development and staging.
- SRE and platform engineers adding local or pre-production SQL observability.
- AI coding agents and contributors that need clear module boundaries and contribution rules.

## Functional Requirements

- Document SQL Lens as a transparent SQL debug proxy, not a database middleware, governance platform, HA proxy, query router, sharding layer, data-sync system, or SQL rewrite engine.
- Prioritize v1 MySQL-compatible protocol support for MySQL, StarRocks, TiDB, and Doris.
- Design protocol, storage, API, UI, and plugin boundaries so future PostgreSQL, SQLite, ClickHouse, and other SQL execution surfaces can be added without rewriting core capture models.
- Cover SQL capture, prepared statement parameter expansion, timeline, replay, slow SQL, error SQL, statistics, dashboard, and web UI.
- Explain default in-memory ring buffer storage, optional SQLite, and future DuckDB support.
- Define REST API, WebSocket API, JSON schema shape, OpenAPI strategy, and error codes.
- Define configuration structure for listen, backend, TLS, logging, storage, retention, auth, web, and proxy behavior.
- Define security design for passwords, TLS, log redaction, SQL redaction, permissions, web login, CSRF, XSS, and RBAC.
- Define plugin extension points such as `OnConnect`, `OnQuery`, `OnPrepare`, `OnExecute`, `OnError`, exporters, webhooks, Prometheus, and OpenTelemetry.
- Define a v0.1 to v2.0 roadmap and a v1.0 milestone breakdown where each task is independently reviewable.
- Generate at least 100 GitHub issue drafts with title, description, acceptance criteria, labels, priority, difficulty, estimated time, and dependencies.
- Define testing strategy for unit tests, protocol tests, benchmarks, integration tests, compatibility tests, Docker Compose, and GitHub Actions.
- Define UI information architecture, layout, interactions, theme, dark mode, responsive behavior, Dashboard, SQL List, SQL Detail, Statistics, Connections, and Settings.

## Non-Functional Requirements

- Documentation must be clear enough for human contributors and AI coding agents to work independently.
- Prefer Rust for backend architecture. Mention Go only as an optional alternative for components where it would be practical.
- Favor KISS, YAGNI, and DRY in all recommendations.
- Keep modules small, explicit, and independently testable.
- Design for maintainability before feature breadth.
- Use English for project documentation to match repository guideline language.
- Avoid promising production traffic governance or database proxy replacement behavior.

## Out of Scope

- Writing backend Rust code.
- Writing frontend React code.
- Creating build scripts, package manifests, Docker files, or CI YAML.
- Implementing MySQL protocol parsing.
- Implementing UI components.
- Creating logos, screenshots, binary assets, or generated images.
- Committing or pushing changes.

## Acceptance Criteria

- [ ] All required documentation files exist at the repository root.
- [ ] `README.md` includes project introduction, motivation, features, screenshot placeholders, installation, quick start, architecture, roadmap, FAQ, contributing, and license.
- [ ] `ARCHITECTURE.md` explains proxy architecture, data flow, network model, threading model, async model, module responsibilities, proxy lifecycle, prepared statement lifecycle, WebSocket push flow, API flow, and storage model.
- [ ] `PRD.md` covers positioning, users, goals, non-goals, roadmap, Dashboard, SQL Detail, Connections, Statistics, Search, Replay, and future plans.
- [ ] `PROTOCOL.md` covers MySQL protocol support, unsupported protocol areas, prepared statement expansion, authentication, packets, commands, sequence IDs, connections, protocol state machine, and future PostgreSQL design.
- [ ] `API.md` defines REST API, WebSocket API, OpenAPI strategy, JSON schemas, and error codes.
- [ ] `CONFIG.md`, `STORAGE.md`, `SECURITY.md`, and `PLUGIN.md` describe their domains with implementation-ready structure and constraints.
- [ ] `ROADMAP.md`, `MILESTONE.md`, and `ISSUES.md` provide an actionable path to v1.0 and beyond.
- [ ] `ISSUES.md` contains at least 100 complete GitHub issue drafts.
- [ ] `TESTING.md` and `BENCHMARK.md` define practical validation and performance goals.
- [ ] `UI.md` defines a usable web application design, not a marketing-only page.
- [ ] `AGENTS.md` gives AI coding agents detailed project structure, module rules, forbidden actions, testing rules, API stability rules, and coordination guidance.
- [ ] No business code is introduced.
- [ ] Existing user changes are not reverted.
